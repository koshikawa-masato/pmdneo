## 1. 中間データの目的と基本方針

### 1.1 PMDにおけるレジスタ寄りの中間データ

PMD（KAJA氏）の音色データ構造は、YM2608のFMレジスタに極めて近い形式で設計されている。

典型的な音色パラメータの並びは以下の通り：

- AL（Algorithm） / FB（Feedback）
- Op1〜Op4の各パラメータ：AR, DR, SR, RR, SL, TL, KS, ML, DT, AM, SSG-EG

これらは、ドライバ内部で**レジスタ書き込み用にほぼそのまま整形された状態**で保持されている。
つまり、PMDの音色データ自体が「FMレジスタコマンドに展開しやすい中間状態」と言える。

### 1.2 過去の「PMD形式を中間データとして使う」事例

2010年代に @Judstyle 氏が関わっていたツール群（F2PCV.dll → PMDRC.dll → PMDRCC.exe）では、以下のようなアプローチを取っていた：

- MMLを一旦 **PMDの演奏データ形式（.M系バイナリ）** にコンパイルする
- このPMD形式自体を「中間データ」として扱う
- その後、PMDデータ → FMP / MXDRV / ZMUSIC などへ変換

この方法の利点は、**すでに多くのツールが存在するPMD形式を再利用** することで、ゼロから中立的な中間言語を作るコストを大幅に削減できる点にある。

### 1.3 MML方言の壁

複数のMML系を扱う上で最大の障壁は、以下の点である：

- ポルタメント、ベンド、LFOの仕様の違い
- ハードエンベロープの扱い
- クロック解像度（#Zenlen / C値）の違い（例: 96 vs 192）
- 音色データをシーケンスデータに埋め込むかどうかの違い

これらのため、「完全に中立で高レベルな中間MML」を作ろうとすると、変換が極めて困難になる。

**示唆**: レジスタコマンド寄りの中間形式を採用することで、上記の方言の違いをある程度吸収しやすくなる。

## 2. ハイブリッド形式（レジスタコマンド + 抽象イベント）

### 2.1 MewMML / MewFMの内部構造

MewMMLPadおよびMewFMは、FMPスタイルを強く意識して設計されている。

- レジスタ直接書き込みコマンド（`y` コマンド系）に対応
- 音色データはYM2608レジスタ値に極めて近い形式で保持
- 内部データは「Operator設定（AR/DR/SR/RR/SL/TL/MUL/DT/AM/SSG-EG/Feedback/Algorithm）」をそのまま記録

これは、MewFMが「VSTiとして現代的に使う」だけでなく、「MMLで低レベル制御したい」という需要にも応える設計であることを示している。

### 2.2 海外ツールでのハイブリッド形式の採用例

**Furnace Tracker**（海外の多チップ対応トラッカー）では、以下の構造を内部で採用している：

- 抽象化された「音色イベント」（Instrument）
- 実際のレジスタ書き込みコマンド（Register Write）
- タイムスタンプ付きのコマンド列

これにより、OPN/OPNA/OPNB/OPMなど異なるチップ間でも、ある程度の互換性を保ちつつ、低レベル制御も可能にしている。

**mml2vgm** でも、レジスタ書き込みをタイムスタンプ付きで記録する形式を取っており、VGM出力やエミュレータ再生に適した中間表現を実現している。

### 2.3 ハイブリッド形式の利点と課題

**利点**:
- 高レベルMMLの差異を吸収しやすい
- 低レベル制御（生レジスタ書き込み）も表現可能
- ドライバ側での解釈コストを調整できる

**課題**:
- 抽象度のバランスをどこで取るか（イベントが多すぎると変換が難しくなる）
- タイムスタンプを付けるかどうかの判断

**示唆**: 最初は「抽象イベントを基本とし、必要に応じてレジスタコマンドに展開する」ハイブリッド形式が、現実的かつ拡張性が高い。

## 3. OPNB / YM2610特有の考慮点

### 3.1 FM3拡張モード（Operator Split）の詳細

OPN（YM2203）およびOPNA（YM2608）では、**FMチャンネル3のオペレータを分割** することで、独立した周波数を持つ3つの音を鳴らすことができる。

**関連レジスタ**:
- `0x27`（ビット6-7）：FM3拡張モードの設定
- `0xA8〜0xAE`：独立F-Number / Block（オペレータ1〜4用）
- `0xB0` 周辺：通常のF-Numberと併用

このモードを有効にすると、通常の4オペレータ和音ではなく、**3つの独立した単純波形** として扱われるため、音色の表現力は大幅に低下するが、複雑なパーカッションや金属音、非整数倍音の表現が可能になる。

### 3.2 NEOGEO実装との関係

NEOGEO AESに搭載されているYM2610は、CH1とCH4がオミットされたバージョンである。

しかし、**FM3拡張モードは実装されており**、コマンドを送ってもスルーされるだけなので、PMD98ドライバの仕様にかなり近づけることができる。

### 3.3 示唆

- 中間データでは、**FM3拡張モードの有無を明示的に記録** できるようにすべき。
- 拡張モード時は音色データも「4オペレータ独立」として扱う必要がある。
- .NEOコンテナ設計では、通常モードと拡張モードを区別したデータ構造を最初から考慮する価値が高い。

## 4. .NEOコンテナとの関係

### 4.1 MewFMのsave state dataから得られる示唆

MewFMのsave state dataは、以下の特徴を持っている：

- 先頭に `VC2!` マジックナンバー + サイズ情報
- その後にXML形式で構造化されたデータ
- `<MewFMState>`、`<FMTone>`、`<Op>` などの要素で音色パラメータを明確に分離

この構造は、.NEOコンテナにおいても「セクション単位で明確に分離する」設計が扱いやすいことを示唆している。

### 4.2 海外ツールの傾向

Furnace TrackerやDefleMaskなどの海外ツールでは、複数のFMチップに対応するため、**チップ非依存の中間表現** を内部で持っている。

これらのツールは、ヘッダー・音色セクション・シーケンスセクションを明確に分離しており、バージョン管理や前方互換性にも配慮している。

### 4.3 示唆

- .NEOコンテナも、ヘッダー・メタ情報・MML中間データ・音色データ・ADPCMデータなどをセクション単位で明確に分離すべき。
- バージョン管理と前方互換性を最初から意識した設計が望ましい。

## 5. 変換元MMLとのマッピング方針

### 5.1 MewMML / MewFMとの親和性

MewFMはFMP/PMD/MUCOMからの音色インポートに正式対応しており、内部データ構造がレジスタ寄りであるため、.NEOの中間データとの相性が良い。

### 5.2 mdxとの関係

@Awed_Urshy氏によって、mdx → MewMML の変換ツールが存在する。
これは、.NEOがMewMMLとmdxの橋渡し役になれる可能性を示している。

### 5.3 示唆

- 最初は **MewMML + PMD** を優先的に対応。
- mdx、MUCOM88、FMP7などは、後から拡張する形で設計する。

## 6. 拡張性と将来性

### 6.1 AES+ / YM2610B対応

AESを改造してYM2610Bを積む、またはAES+をハックしてYM2610Bモードで動作させるという野心的なアプローチを視野に入れる場合、中間データは以下の点を考慮する必要がある：

- YM2610（オミット版）とYM2610Bの違いを吸収
- FM3拡張モードのさらなる活用
- ADPCM-Aのフル活用

### 6.2 示唆

- FM3拡張やADPCM-A/Bの扱いを、最初から拡張可能な形で設計しておく。
- ハードウェア情報（YM2610 / YM2610B）を記録できるフィールドを用意する。

## 7. 検討事項セクションで埋めるべきポイント

以下の点は、特に他エージェントが検討しやすいよう、具体的に列挙する。

- 中間データのバイナリ形式 vs テキスト形式の選択
- イベントの抽象度をどこまで高めるか（レジスタ直叩きとのバランス）
- FM3拡張モード時の音色データの扱い方（4オペレータ独立 vs 通常音色）
- ADPCMサンプルの圧縮形式の標準化（18.5kHz ADPCM-Aなど）
- クロック解像度（Zenlen/C値）の違いの吸収方法
- タイムスタンプ付きレジスタコマンド形式の必要性
- .NEOコンテナ全体のバージョン戦略と前方互換性
