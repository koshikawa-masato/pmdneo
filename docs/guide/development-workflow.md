# 開発の進め方

この文書は、PMDNEO の開発で使う基本的な流れを説明します。

PMDNEO では、思い込みで実装に入らないことを重視しています。まず仕様を言葉にし、その後で実装と検証を行います。

## 基本の流れ

```text
論点を整理する
  |
  v
設計書または ADR で方針を固定する
  |
  v
小さく実装する
  |
  v
機械的に検証する
  |
  v
MAME で動作を確認する
  |
  v
必要なら人が試聴する
  |
  v
結果を ADR や handoff に残す
```

## 設計書ファースト

PMDNEO では、実装前に設計を文書化します。

これは、過去に「動いたつもり」や「互換だと思い込んだ」状態で進んでしまった反省から来ています。

実装前に確認すること:

- 何を実装するか
- 何を今回は実装しないか
- 既存形式や既存ドライバとの関係
- どう検証するか
- 完了条件は何か

## ADR の使い方

設計判断は ADR に残します。

PMDNEO の ADR は、短い意思決定記録だけでなく、AI と開発を続けるための作業記録も兼ねます。

そのため、進行中の ADR には次を書いてよいです。

- 壁打ち結果
- 採用案と不採用案
- 実装範囲
- scope-out
- 検証方法
- 完了判定
- 後続作業

## 実装単位

実装は、小さな単位に分けます。

PMDNEO では、よく `step` や `sub-sprint` という単位で進めます。

小さく分ける理由:

- どこで壊れたか分かりやすい
- 検証しやすい
- 判断を途中で修正しやすい
- AI が文脈を追いやすい

## 検証

PMDNEO では、ビルド成功だけでは完了にしません。

検証には複数の層があります。

| 検証 | 目的 |
|---|---|
| ビルド確認 | ROM や中間ファイルが作れることを確認する |
| byte 比較 | 期待するバイナリと一致するか確認する |
| trace 確認 | レジスタ書き込みや実行経路を確認する |
| MAME 録音 | 実際に音が出ているか確認する |
| 人の試聴 | 機械判定しにくい音の違いを確認する |

詳しくは [検証ガイド](verification-guide.md) を参照してください。

## handoff の役割

handoff は、作業の引き継ぎ資料です。

ADR が「何を決めたか」を中心に残すのに対して、handoff は「何を実施し、どう確認したか」を詳しく残します。

PMDNEO では、handoff も AI協働に必要な記録です。勝手に削除したり短縮したりしません。

## docs 整理と実装は分ける

ドキュメント整理と実装変更は、できるだけ別作業にします。

理由:

- 差分の目的が分かりやすい
- レビューしやすい
- 既存ADRを誤って壊しにくい
- 実装の検証と文書整理の確認を分けられる

人間向けdocsを作る場合も、既存ADRは元資料として残します。
