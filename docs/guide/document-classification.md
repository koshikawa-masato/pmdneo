# ドキュメント分類ルール

PMDNEO の資料は、読む目的ごとに分けます。

## 公開 docs に置くもの

`docs/` には、人が読んで理解するための資料を置きます。

対象:

- 現在有効な仕様
- 現在有効な設計
- 開発者が再現できる手順
- 意思決定の要約
- 外部の人にも説明できる背景

文章は短く、平易にします。AI 向けの圧縮表現は避けます。

## AI協働用ADRに置いてよいもの

`docs/adr/00xx-*.md` は、AI協働用の高密度な作業台でもあります。

そのため、進行中の開発では次を ADR に含めてよいです。

- AI との壁打ち結果
- 採用案と不採用案
- scope-in / scope-out
- verify gate
- audio gate
- trace 結果
- commit ごとの作業記録
- 後続AIが文脈を復元するための詳細

この運用は今後も継続します。人間向けdocsを整えることは、AI協働用ADRを止めることではありません。

## 内部資料に置くもの

`internal-docs/` には、公開資料として読むには重すぎる作業記録を置きます。

対象:

- AI との対話を続けるための高密度メモ
- 長い verify log
- MAME trace や register trace の詳細
- commit ごとの作業報告
- 調査途中の断片メモ
- 公開 docs へ移す前の下書き

`internal-docs/` は git 管理外です。必要な内容は要約して `docs/` に移します。

## 既存 ADR の扱い

既存の ADR には、意思決定、実装ログ、検証証跡、AI 向け圧縮メモが混ざっています。

整理するときは、削除ではなく分離を基本にします。

| 内容 | 移動先 |
|---|---|
| 決定の要約 | `docs/adr/` |
| 現在有効な仕様 | `docs/design/` または `docs/spec/` |
| 開発手順 | `docs/guide/` |
| 長い検証証跡 | AI協働用ADR、handoff、または `internal-docs/` |
| 作業履歴 | AI協働用ADR、handoff、または `internal-docs/` |
| 未整理メモ | `internal-docs/` |

## 移行時の判断基準

迷ったら、次の質問で分けます。

- 初めて読む人が必要とするか
- 現在の仕様を理解するために必要か
- 後から検証するときだけ必要か
- AI に長い文脈を渡すためだけに必要か

最初の2つが「はい」なら、人間向け `docs/` に残します。

後ろの2つが「はい」なら、AI協働用ADR、handoff、または `internal-docs/` に残します。既存ADRを勝手に移動するという意味ではありません。

## 先に整理する候補

最初の移行対象は、ADR-0026 から ADR-0033 です。

理由:

- 1ファイルが長い
- 同じ構造の繰り返しが多い
- PMDNEO 固有の検証ログが多い
- 公開資料としては要約版が必要

まず1本だけ人間向けの要約を別ファイルとして作り、その型を他の要約文書に広げます。元の ADR は保持します。
