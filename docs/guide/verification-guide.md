# 検証ガイド

この文書は、PMDNEO で使う検証の考え方を説明します。

PMDNEO は音を扱うプロジェクトです。そのため、ビルドが通るだけでは十分ではありません。バイナリ、実行経路、レジスタ書き込み、実際の音を分けて確認します。

## 検証の層

PMDNEO の検証は、主に次の層に分かれます。

| 層 | 目的 |
|---|---|
| ビルド確認 | ファイルを生成できるか確認する |
| byte 比較 | 出力が期待値と一致するか確認する |
| trace 確認 | 実行経路やレジスタ書き込みを確認する |
| MAME 録音 | 実際に音が出るか確認する |
| 試聴 | 人が聞いて意味のある違いを確認する |

どれか1つだけで十分とは考えません。変更内容に応じて、必要な層を選びます。

## ビルド確認

ビルド確認は、最初の検証です。

ROM、曲データ、サンプルデータなどが生成できるかを確認します。

ただし、ビルド成功は「動く」ことの証明ではありません。

## byte 比較

byte 比較は、既存出力を壊していないか確認するために使います。

たとえば、既存 `.M` 出力を変えないことが重要な場合、改修前後の出力を byte 単位で比較します。

この検証は、互換性を守るために重要です。

## trace 確認

trace 確認では、実行中に何が起きたかを見ます。

PMDNEO では、特に次を確認することがあります。

- 期待した routine に到達したか
- 期待したレジスタへ書き込んだか
- Kパートとメロディ内リズム命令が同じ経路に合流したか
- サンプルアドレスが意図通り選ばれたか

trace は、音が聞こえた理由を説明するための証拠になります。

## MAME 録音

PMDNEO の実行確認には MAME を使います。

MAME で ROM を起動し、音声を録音します。録音結果は、無音でないか、期待した音が出ているか、比較対象と同じかを確認するために使います。

ただし、録音ファイルの完全一致だけに頼りすぎないようにします。タイミング差や環境差で、音としては同じでもファイルの hash が変わることがあります。

## 試聴

音の判断には、人の試聴が必要な場合があります。

たとえば、次のような確認です。

- Kパートとメロディ内リズム命令が同じ音に聞こえるか
- BD、SD、HH、CYM、TOM、RIM が区別できるか
- 同時トリガー時に期待した音の支配関係になっているか

試聴結果は、ADR や handoff に記録します。

## verify script

`src/test-fixtures/` には、step ごとの検証用 MML と verify script があります。

例:

- `src/test-fixtures/step12/`
- `src/test-fixtures/step13/`
- `src/test-fixtures/step17/`
- `src/test-fixtures/step18/`

これらは、過去の判断を壊していないか確認するための重要な資産です。

## audio gate

audio gate は、実際に音を確認するための手順です。

PMDNEO では、機械的な検証と人の試聴を分けて扱います。

機械的な検証:

- byte 比較
- trace 確認
- register write 確認
- script の PASS / FAIL

人の試聴:

- 音が出ているか
- 音色が区別できるか
- 同じ経路の音が同じに聞こえるか

どちらも重要です。

## 完了判定

完了判定は、作業ごとに ADR で先に決めます。

良い完了判定には、次が含まれます。

- 何を確認するか
- どの script を通すか
- MAME で確認するか
- 試聴が必要か
- 何を今回は確認しないか

完了判定が曖昧なまま実装に入ると、後で「何をもって完了か」が分からなくなります。

## 注意点

検証ログは長くなりがちです。

長い trace や script 出力は、人間向けdocsの本文にそのまま貼らず、ADR、handoff、内部資料に分けます。

人間向けdocsでは、検証の目的と結果を短く説明し、必要な詳細資料へリンクします。
