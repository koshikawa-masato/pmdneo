# ADR の読み方

PMDNEO の ADR は、一般的な短い意思決定記録よりも長いものがあります。

これは、Claude Code、ChatGPT、Codex と開発を継続するために、判断経緯や検証結果を細かく残しているためです。

## ADR は2つの役割を持つ

PMDNEO の ADR には、次の2つの役割があります。

| 役割 | 説明 |
|---|---|
| 意思決定記録 | 何を決めたか、なぜそうしたかを残す |
| AI協働ログ | 後続のAIが文脈を復元できるよう、壁打ちや検証結果を残す |

そのため、特に ADR-0026 以降は長く、高密度です。

これは失敗ではありません。PMDNEO の開発を進めるために必要だった記録です。

## 最初に読む場所

長い ADR を読むときは、最初から最後まで通読しようとしないでください。

まず次を確認します。

1. 冒頭のタイトル
2. 状態
3. 背景の節
4. 決定の節
5. scope-in / scope-out の節
6. 完了判定の節

Annex、trace、commit 列、verify gate の詳細は、必要になったときだけ読みます。

## 状態の意味

ADR の状態は、おおむね次の意味です。

| 状態 | 意味 |
|---|---|
| Draft | 方針を固めている途中 |
| Accepted | その判断を採用済み |
| scope-out | 今回は扱わないと明示したもの |

PMDNEO では、Accepted になった ADR でも、詳細な作業ログが残っていることがあります。

## ADR-0026 以降について

ADR-0026 以降には、K/R リズム互換、ドラム種別拡張、同時トリガー、サンプル由来管理などの詳細な記録があります。

これらは、AI協働用の高密度 ADR です。

読むときは、次のように考えると分かりやすくなります。

| ADR 群 | 主な内容 |
|---|---|
| ADR-0026 | K/R リズム互換の基礎 |
| ADR-0027〜0031 | BD、SD、HH、CYM、TOM、RIM の段階的対応 |
| ADR-0032 | 複数リズム同時トリガーの意味付け |
| ADR-0033 | リズム音源サンプルの由来と自作化方針 |

詳細な証跡が必要ない場合は、各 ADR の「決定」と「完了判定」を中心に読んでください。

## AI協働用ADRは継続する

PMDNEO では、今後も AI協働用ADR を作成します。

これは、人間向けdocsとは役割が違います。

| 種類 | 目的 |
|---|---|
| AI協働用ADR | 開発を止めずに進めるための作業台 |
| 人間向けdocs | 第三者が理解するための入口 |

AI協働用ADRを無理に短くすると、後続のAIが文脈を復元できなくなります。

そのため、既存ADRは元資料として保持します。人間向けに説明したい場合は、別のガイド文書として要約します。

## 書き換え時の注意

既存ADRを整理する場合は、必ずユーザー確認が必要です。

勝手にしてはいけないこと:

- 既存ADRを削除する
- 既存ADRを短縮する
- 既存ADRを人間向け文体へ全面変換する
- handoff や trace 記録を消す

公開docsが必要な場合は、元ADRを残したまま、別ファイルで要約します。
