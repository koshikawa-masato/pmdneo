# 人が読むためのドキュメント方針

PMDNEO の初期ドキュメントには、AI と長い文脈を共有するための圧縮表現が多く含まれています。

この形式は、AI協働用ADRとして今後も使います。ただし、人間向けの公開資料では、その形式をそのまま使いません。

## 基本方針

- 結論を先に書く
- 1文に複数の判断を詰め込まない
- `=`, `literal`, `scope-out`, `gate`, `sprint chain` などの内部用語を使いすぎない
- 必要な場合は、最初に普通の日本語で説明してから専門用語を添える
- 検証ログと意思決定を同じ段落に混ぜない

## 書き換え例

避けたい書き方:

```md
Step 17 は K/R drum kind expansion proof — i = RIM single-kind / dispatch path 1 本化不変 / full 6 drum completion milestone。
```

人向けの書き方:

```md
Step 17 では、PMD の `\i` 命令を RIM として再生できるようにした。

これで BD、SD、CYM、HH、TOM、RIM の6種類すべてのリズム音が PMDNEO で使えるようになった。
Kパートとメロディ内のリズム命令は、同じ再生処理に合流する。
```

## 人間向け要約の短い型

```md
# 短い題名

## 概要

何を決めたかを2〜5文で書く。

## 背景

なぜこの判断が必要になったかを書く。

## 決定

- 決めたこと
- 変えないこと
- 後回しにすること

## 理由

採用理由を書く。

## 影響

実装、テスト、利用者への影響を書く。

## 詳細資料

- 元になった ADR や handoff への参照
```

## 内部資料へ逃がすもの

- 長い trace
- verify script の全出力
- commit ごとの細かい作業ログ
- AI との壁打ち記録
- 未整理の調査メモ
- 公開資料として読む必要がない一時判断

これらは失ってはいけない場合があります。AI協働用ADRや handoff には残してよいです。

人間向けdocsでは、必要な内容だけを短く要約し、詳細資料へリンクします。
